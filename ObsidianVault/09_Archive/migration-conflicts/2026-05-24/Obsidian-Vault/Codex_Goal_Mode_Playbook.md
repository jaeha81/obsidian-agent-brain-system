# Codex `/goal` 모드 운영 원칙 및 지식 통합 시스템 적용 자료

## 1. 이 문서의 목적

이 문서는 사용자가 제공한 유튜브 영상 내용을 바탕으로, 현재 운영 중인 **Obsidian + Claude Code + Codex 기반 세컨드 브레인/개발 통합 시스템**에 적용하기 위한 운영 지침이다.

핵심 목적은 다음과 같다.

1. Codex의 `/goal` 모드를 무한 반복이나 조기 종료 없이 안정적으로 사용한다.
2. Obsidian Vault를 외부 메모리로 활용해 Claude Code와 Codex가 같은 맥락을 공유하게 한다.
3. 모든 개발, 자동화, 시스템 개선 작업을 `목표 → 실행 → 검증 → 기록 → 개선` 루프로 관리한다.
4. 영상, 글, 강의, 자료에서 얻은 지식을 단순 요약이 아니라 실제 시스템 운영 규칙으로 전환한다.

---

# 2. 핵심 요약

Codex의 `/goal` 모드는 일반적인 단발성 프롬프트가 아니라 **목표 달성형 반복 작업 모드**로 이해해야 한다.

즉, 에이전트가 다음 과정을 반복한다.

```text
목표 확인
→ 작업 실행
→ 결과 검증
→ 목표 충족 여부 판단
→ 부족하면 다시 수정
→ 목표 충족 시 종료
```

따라서 `/goal` 모드에서 가장 중요한 것은 **언제 멈출지 알 수 있는 종료 조건**이다.

종료 조건이 불명확하면 두 가지 문제가 발생한다.

| 실패 유형 | 설명 |
|---|---|
| Exit Too Early | 몇 분 작업하고 “더 할 게 없다”고 판단해 너무 빨리 종료 |
| Never Stop | 목표가 모호해서 끝없이 수정하며 멈추지 않음 |

결론적으로 `/goal` 모드는 다음 3가지가 반드시 필요하다.

1. **명확하고 측정 가능한 목표**
2. **짧고 신뢰할 수 있는 피드백 루프**
3. **Obsidian 기반 외부 메모리**

# 3. 우리 통합 시스템에 적용하는 방식

현재 시스템 구조는 다음과 같이 해석한다.

```text
사용자 입력 자료
→ ChatGPT가 핵심 구조화
→ Obsidian Vault에 Markdown 저장
→ Claude Code / Codex가 Markdown 읽기
→ 개발, 자동화, 정리, 개선 작업 수행
→ 결과를 다시 Obsidian에 기록
```

앞으로 유튜브 영상, 블로그 글, 개발 아이디어, 자동화 전략, 사업 아이디어가 들어오면 단순 메모가 아니라 아래 형태로 시스템화한다.

```text
정보 수집
→ 핵심 원칙 추출
→ 적용 가능한 운영 규칙으로 변환
→ Obsidian에 저장
→ Codex/Claude Code 작업 지침으로 연결
→ 실험 결과 기록
```

## 각 파일의 역할

| 파일 | 역할 |
|---|---|
| `plan.md` | 상위 계획, 목표, 방향성, 초기 전략 기록 |
| `experiment.md` | 무엇을 시도했고 어떤 결과가 나왔는지 기록하는 핵심 파일 |
| `notes.md` | 작업 중 떠오른 생각, 가정, 임시 메모 기록 |
| `Codex_Goal_Mode_Playbook.md` | Codex `/goal` 운영 원칙을 정리한 기준 문서 |
| `goal_based_development.md` | 실제 개발 작업에 적용할 워크플로우 문서 |

가장 중요한 파일은 `experiment.md`다.

이 파일은 사람과 에이전트가 함께 보는 **단일 진실 공급원, Single Source of Truth** 역할을 한다.

# 5. Codex `/goal` 모드 사용 원칙

## 원칙 1. 목표는 반드시 측정 가능하게 만든다

나쁜 예시는 다음과 같다.

```text
내 코드를 더 좋게 만들어줘.
시스템을 개선해줘.
자동화 구조를 최적화해줘.
```

이런 요청은 종료 조건이 없기 때문에 `/goal` 모드에서 위험하다.

좋은 예시는 다음과 같다.

```text
특정 파일의 실행 시간을 기존 대비 20% 단축하라.
기존 단위 테스트와 통합 테스트가 모두 통과해야 한다.
변경한 내용은 experiment.md에 기록하라.
목표 달성 여부를 테스트 결과로 판단하라.
```

## 목표 정의 템플릿

```text
목표:
- [구체적인 작업 대상]을 [측정 가능한 기준]까지 개선한다.

기준:
- 현재 상태: [baseline]
- 목표 상태: [target]
- 검증 방법: [test / script / checklist / benchmark]
- 종료 조건: [성공 기준]
- 제약 조건: [하지 말아야 할 것 / 유지해야 할 것]
```

# 6. 정성적 목표를 체크리스트로 바꾸는 방법

정성적인 목표도 체크리스트로 바꾸면 `/goal` 모드에서 사용할 수 있다.

예를 들어 “문서를 더 좋게 정리해줘”는 모호하다.

이를 다음처럼 바꾼다.

```markdown
## 종료 체크리스트

- [ ] 문서 제목이 명확하다.
- [ ] 핵심 요약이 5줄 이내로 정리되어 있다.
- [ ] 적용 방법이 단계별로 정리되어 있다.
- [ ] Codex 실행 프롬프트가 포함되어 있다.
- [ ] Obsidian 저장 위치가 명시되어 있다.
- [ ] 실험 기록 방식이 포함되어 있다.
- [ ] 다음 액션이 명확하다.
```

이렇게 하면 Codex는 “좋은 문서”라는 추상적 기준이 아니라 체크리스트 완료 여부로 종료할 수 있다.

# 7. 짧은 피드백 루프 설계

`/goal` 모드는 반복 실행 구조이므로, 피드백이 느리면 효율이 떨어진다.

좋은 피드백 루프는 다음 조건을 가진다.

1. 빠르게 실행된다.
2. 결과가 명확하다.
3. 목표 달성 여부를 판단할 수 있다.
4. 전체 작업을 대표할 만큼 신뢰할 수 있다.

## 개발 작업 예시

나쁜 방식:

```text
전체 시스템을 매번 완전히 빌드하고 전체 테스트를 돌린다.
```

좋은 방식:

```text
우선 핵심 모듈 테스트만 빠르게 실행한다.
문제가 없으면 이후 전체 테스트를 실행한다.
```

## 문서 정리 작업 예시

나쁜 방식:

```text
문서가 괜찮아질 때까지 계속 수정한다.
```

좋은 방식:

```text
체크리스트 10개 중 10개가 충족되면 종료한다.
```

# 8. 외부 메모리 운영 방식

Codex와 Claude Code가 장시간 작업할 때 모든 맥락을 대화창 안에만 두면 일관성이 떨어질 수 있다.

따라서 중요한 작업 맥락은 반드시 Markdown 파일에 기록한다.

권장 파일은 다음 3개다.

```text
plan.md
experiment.md
notes.md
```

## `plan.md` 템플릿

```markdown
# plan.md

## 작업 목표

## 현재 상태

## 목표 상태

## 핵심 전략

## 작업 범위

## 제외 범위

## 성공 기준

## 예상 리스크

## 다음 단계
```

## `experiment.md` 템플릿

```markdown
# experiment.md

## 실험 목적

## 기준 지표

| 항목 | 현재값 | 목표값 | 측정 방법 |
|---|---:|---:|---|
| 예: 실행 시간 | 10초 | 8초 이하 | benchmark script |

## 실험 기록

| 날짜 | 시도 내용 | 결과 | 판단 | 다음 액션 |
|---|---|---|---|---|
| 2026-05-15 | 초기 구조 분석 | 병목 후보 발견 | 추가 테스트 필요 | 핵심 함수 벤치마크 |

## 최종 결론

## 반영된 변경사항

## 남은 과제
```

## `notes.md` 템플릿

```markdown
# notes.md

## 실시간 메모

### 2026-05-15

- 가정:
- 관찰:
- 의문:
- 다음에 확인할 것:
```

# 9. Codex에 줄 수 있는 `/goal` 실행 프롬프트

아래 프롬프트는 Codex에서 `/goal` 모드를 켠 뒤 사용할 수 있는 기본형이다.

```text
/goal

목표:
Obsidian Vault에 저장된 다음 Markdown 문서를 읽고, 현재 지식 통합 시스템에 반영하라.

대상 문서:
- 00_SYSTEM/AI_Integration/Codex_Goal_Mode_Playbook.md
- 01_EXPERIMENTS/codex_goal_mode/plan.md
- 01_EXPERIMENTS/codex_goal_mode/experiment.md
- 01_EXPERIMENTS/codex_goal_mode/notes.md

수행 작업:
1. 문서의 핵심 원칙을 분석한다.
2. 현재 프로젝트의 개발/자동화/지식관리 워크플로우에 적용 가능한 항목을 추출한다.
3. 적용할 수 있는 규칙과 명령 템플릿을 정리한다.
4. 필요한 경우 프로젝트 내 관련 문서를 업데이트한다.
5. 수행한 작업과 판단 근거를 experiment.md에 기록한다.

종료 조건:
- [ ] `/goal` 모드 운영 원칙이 시스템 문서에 반영되어 있다.
- [ ] 명확한 목표 설정 템플릿이 추가되어 있다.
- [ ] 피드백 루프 설계 원칙이 추가되어 있다.
- [ ] plan.md / experiment.md / notes.md의 역할이 정리되어 있다.
- [ ] 실제 작업에 사용할 Codex 프롬프트 예시가 추가되어 있다.
- [ ] 변경 내용이 experiment.md에 기록되어 있다.

제약 조건:
- 기존 시스템 구조를 임의로 삭제하지 말 것.
- 변경 전후 내용을 구분해서 기록할 것.
- 모호한 목표는 체크리스트로 변환할 것.
- 완료 여부는 체크리스트 기준으로 판단할 것.
```

# 10. Claude Code에 줄 수 있는 협업 지침

Claude Code에는 `/goal` 명령이 Codex와 동일하게 작동하지 않을 수 있으므로, 아래처럼 목표 지향형 작업 지침으로 준다.

```text
너는 현재 Obsidian Vault 기반 세컨드 브레인/개발 통합 시스템의 협업 에이전트다.

작업 방식:
1. 사용자의 요청을 명확한 목표로 변환한다.
2. 목표가 모호하면 체크리스트로 바꾼다.
3. 작업 전 plan.md를 확인한다.
4. 작업 중 notes.md에 중요한 판단과 가정을 기록한다.
5. 작업 후 experiment.md에 시도 내용, 결과, 다음 액션을 기록한다.
6. 종료 조건이 충족되기 전까지 임의로 완료 판단하지 않는다.

반드시 지킬 원칙:
- 목표는 측정 가능해야 한다.
- 피드백 루프는 짧아야 한다.
- 모든 중요한 작업 결과는 Markdown에 기록해야 한다.
- experiment.md를 단일 진실 공급원으로 사용해야 한다.
```

# 11. 앞으로 유튜브 영상 자료를 시스템에 넣는 표준 프로세스

앞으로 사용자가 유튜브 영상 내용, 블로그 글, 강의 요약, 개발 팁을 붙여 넣으면 다음 방식으로 처리한다.

```text
1. 원문 입력
2. 핵심 개념 추출
3. 시스템 적용 가능성 분석
4. Obsidian용 Markdown 문서 생성
5. Codex/Claude Code 실행 프롬프트 생성
6. experiment.md 기록 항목 생성
7. 다음 액션 도출
```

## 입력 자료 변환 기준

| 입력 자료 | 변환 결과 |
|---|---|
| 유튜브 영상 스크립트 | 핵심 원칙, 적용 전략, 실행 프롬프트 |
| 블로그 글 | 개념 요약, 체크리스트, 워크플로우 |
| 개발 팁 | 코드 적용 지침, 테스트 조건 |
| 사업 아이디어 | 실행 플랜, 자동화 흐름, 수익화 가설 |
| 자동화 정보 | Make.com/AI Agent 워크플로우 설계 |

# 12. 실제 적용 예시

## 예시 1. 개발 최적화 작업

```text
/goal

목표:
src/main.py의 데이터 처리 속도를 기존 대비 20% 개선하라.

기준:
- 현재 평균 실행 시간: benchmark 기준 10초
- 목표 실행 시간: 8초 이하
- 기존 테스트는 모두 통과해야 함

피드백 루프:
- 우선 benchmark_small.py를 실행한다.
- 목표 달성 후보가 나오면 전체 테스트를 실행한다.

외부 메모리:
- 시도 내용은 01_EXPERIMENTS/performance/experiment.md에 기록한다.

종료 조건:
- [ ] benchmark_small.py 기준 8초 이하 달성
- [ ] 전체 테스트 통과
- [ ] 변경 파일 목록 기록
- [ ] experiment.md에 결과 기록
```

## 예시 3. 유튜브 영상 지식 반영 작업

```text

03_INPUTS/youtube_notes/에 저장된 영상 요약 문서를 읽고, 현재 AI 통합 시스템 운영 규칙에 반영하라.

3. 새롭게 적용할 내용은 00_SYSTEM/AI_Integration/에 추가한다.
```

# 15. 핵심 결론

Codex `/goal` 모드는 강력하지만, 모호한 목표를 주면 제대로 작동하지 않는다.

우리 시스템에서는 앞으로 모든 목표형 작업을 다음 공식으로 처리한다.

```text
명확한 목표
+ 측정 가능한 종료 조건
+ 짧은 피드백 루프
+ Obsidian 외부 메모리
+ experiment.md 기록
= 안정적인 AI 협업 시스템
```

이 구조를 기준으로 하면 Codex와 Claude Code는 단순 응답 도구가 아니라, Obsidian Vault를 중심으로 장기 프로젝트를 함께 운영하는 협업 시스템으로 작동할 수 있다.
