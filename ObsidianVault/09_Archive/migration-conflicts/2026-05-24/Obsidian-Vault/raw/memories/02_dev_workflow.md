## 🔄 핵심 워크플로우

### 표준 구현 프로세스
```
research.md 작성
    ↓
plan.md 작성 (설계·의사코드·트레이드오프)
    ↓
재하의 명시적 승인 ("구현해")
    ↓
구현 (코드 작성)
```

- **승인 전 실코드 생성 금지** (설계·의사코드·스니펫은 승인 없이 제공 가능)
- 작업 규모 분류:
  - **소**: 즉시 실행
  - **중**: 계획 → 승인
  - **대**: 리서치 → 계획 → 승인
  - 보안·결제·인증은 규모 무관 항상 계획 → 승인

### 계획 문서 형식 (중·대규모)
```
구현 방식
수정 파일
Before·After 스니펫
트레이드오프
Todo
```
마지막 문구: "계획이 완료되었습니다. 검토 후 승인해주세요. 아직 코드를 수정하지 않았습니다."

## 🖥️ tmux 5-에이전트 레이아웃

```
┌──────────────┬──────────────┐
│  RESEARCH    │  PLAN        │
│  (정보수집)  │  (설계)      │
├──────┬───────┴──────┬───────┤
│ FE   │   BACKEND    │  QA   │
│      │              │       │
└──────┴──────────────┴───────┘
```

- **RESEARCH**: 기술 조사, 라이브러리 비교, 공식 문서 확인
- **PLAN**: 아키텍처 설계, 파일 구조, 인터페이스 정의
- **FRONTEND**: Next.js 14, React 컴포넌트
- **BACKEND**: FastAPI, 에이전트 로직, DB
- **QA**: 테스트, 통합 검증, 버그 확인

## 🎯 Codex Goal Mode 운영

### 목적
Codex `/goal` 모드는 긴 작업을 `목표 → 실행 → 검증 → 기록 → 개선` 루프로 관리하기 위한 목표형 작업 방식이다.

### 목표 정의 템플릿

```text
목표:
- [작업 대상]을 [측정 가능한 기준]까지 개선한다.

기준:
- 현재 상태:
- 목표 상태:
- 검증 방법:
- 종료 조건:
- 제약 조건:
```

### 정성적 요청 변환 규칙

모호한 요청은 실행 전에 체크리스트로 바꾼다.

```markdown
- [ ] 목표가 구체적이다.
- [ ] 종료 조건이 측정 가능하다.
- [ ] 검증 방법이 있다.
- [ ] 짧은 피드백 루프가 있다.
- [ ] 결과가 sessions/codex-goal-mode-YYYYMMDD.md에 기록된다.
```

### 짧은 피드백 루프

- 개발: 핵심 단위 테스트 또는 작은 벤치마크 → 전체 테스트
- 문서: 종료 체크리스트 검수 → 링크/파일 존재 확인
- 자동화: dry-run 또는 단일 샘플 실행 → 전체 실행

### 외부 메모리

Goal Mode 실행 기록은 기존 Vault 흐름에 맞춰 `sessions/codex-goal-mode-YYYYMMDD.md`에 남긴다. 이 파일이 해당 날짜 Goal Mode 실험의 `experiment.md` 역할을 한다.
