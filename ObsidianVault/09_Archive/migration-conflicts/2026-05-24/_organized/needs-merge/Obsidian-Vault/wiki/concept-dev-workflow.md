# JH 개발 워크플로우

JH 생태계 전체에 적용되는 0~8단계 개발 프로세스.
핵심 원칙: **계획 모드 우선 (Plan First)** — "일단 만들고 수정" 방식 금지.

---

## 단계별 개요

| 단계 | 담당 | 산출물 | 트리거 |
|------|------|--------|--------|
| 0. 기획 검증 | AI | 6질문 확인 | "새 프로젝트", "전면 개편" |
| 1. 리서치 | AI | research.md | 자동 |
| 2. 계획 | AI | plan.md | 자동 |
| 3. 메모 검토 | 사용자 | 인라인 메모 | 사용자 주도 |
| 4. 구현 | AI | 실제 코드 | "구현해", "시작해" |
| 5. 테스트 | AI | 테스트 결과 | 구현 완료 후 |
| 6. 배포 | AI | 배포 결과 | 테스트 통과 후 |
| 7. PR | AI | PR 생성 | 배포 후 |
| 8. 회고 | AI+사용자 | 세션 로그 | 세션 종료 전 |

## 규모별 워크플로우 분기

| 규모 | 기준 | 처리 |
|------|------|------|
| **소** | 단일 파일, 버그픽스, 오타 수정 | 즉시 구현 |
| **중** | 새 기능, API 추가, 2~5개 파일 수정 | plan.md → 승인 → 구현 |
| **대** | 아키텍처 변경, DB 스키마 변경 | 리서치 → plan.md → 승인 → 구현 |

> 규모 판단 불확실 → "중" 기준으로 처리

## Plan First 원칙 (불변)

- 2단계(계획) 완료 및 승인 없이 4단계(구현) 진입 금지
- plan.md 마지막 문구 필수:
  ```
  계획이 완료되었습니다. 검토 후 메모를 남겨주시거나 구현 승인을 해주세요.
  아직 코드를 수정하지는 않았습니다.
  ```

## 자가 검증 피드백 루프 (Boris 원칙)

구현 시작 전 검증 수단 확보 필수:
- 실제 서버 띄우기, 브라우저 확인, curl/HTTP 재현 스크립트
- 수단 없으면 **먼저 검증 인프라 구축** 후 본 구현 시작
- 프론트엔드 작업: 브라우저 실제 동작 확인 후에만 "완료" 보고

## Codex Goal Mode 원칙

`/goal` 모드는 기존 Plan First 원칙을 대체하지 않고, 목표형 작업의 종료 조건을 명확히 하는 보강 규칙으로 사용한다.

### 적용 공식

```text
명확한 목표
+ 측정 가능한 종료 조건
+ 짧은 피드백 루프
+ Obsidian 외부 메모리
+ sessions/codex-goal-mode-YYYYMMDD.md 기록
= 안정적인 AI 협업 시스템
```

### 사용 기준

- 목표가 모호하면 먼저 체크리스트로 변환한다.
- 종료 조건은 테스트, 스크립트, 검수 체크리스트, 파일 존재 여부처럼 확인 가능해야 한다.
- 피드백 루프는 가장 짧은 검증부터 시작하고, 통과 후 더 넓은 검증으로 확장한다.
- 결과와 판단 근거는 해당 날짜의 `sessions/codex-goal-mode-YYYYMMDD.md`에 기록한다.

### Plan First와의 관계

- 소규모 작업: 즉시 실행 가능하되 종료 체크리스트를 둔다.
- 중/대규모 작업: 기존처럼 plan.md 또는 계획 승인 후 구현한다.
- `/goal`은 무한 루프와 조기 종료를 막기 위한 운영 방식이며, 승인 없는 구현을 허용하는 예외가 아니다.
