# LLM Wiki — 복리 지식 베이스 패턴

LLM이 직접 위키를 작성·유지함으로써 지식이 매 세션마다 복리처럼 누적되는 퍼스널 노리지 베이스 패턴. 기존 RAG의 "매번 재발견" 문제를 해소한다.

## 핵심 통찰

**RAG와의 차이:**
- RAG: 원본 파일 → 쿼리 시 청크 검색 → 답변 생성. 누적 없음.
- LLM Wiki: 소스 읽기 → 위키 페이지 작성/갱신 → 교차 참조 → 복리 누적.

**왜 작동하는가:**
- 인간이 위키를 버리는 이유 = 유지 관리 부담. LLM은 지치지 않는다.
- 교차 참조, 모순 플래그, 합성이 이미 완료된 상태에서 쿼리함.
- 좋은 쿼리 답변도 위키 페이지가 되어 다시 복리에 편입.

## 3계층 아키텍처

```
Raw Sources (불변) → Wiki (LLM 소유) → Schema (운영 지침)
```

| 계층 | 소유자 | 역할 |
|------|--------|------|
| Raw Sources | 인간이 추가 | 불변 원본. 진실의 원천. |
| Wiki | LLM이 작성 | 합성·요약·교차 참조된 지식 |
| Schema | 협의로 진화 | LLM 행동 지침 (CLAUDE.md) |

## 3가지 오퍼레이션

### Ingest
소스 추가 시: 읽기 → 논의 → 요약 페이지 생성 → 기존 페이지 업데이트 → 인덱스·로그 갱신.
단일 소스가 10~15개 위키 페이지를 터치할 수 있다.

### Query
질문 시: 인덱스 탐색 → 관련 페이지 읽기 → 답변 합성. 
좋은 답변은 새 페이지로 저장 → 지식 베이스에 편입.

### Lint
주기적 건강 검진: 모순·오래된 클레임·고아 페이지·누락 교차 참조·데이터 갭 점검.

## 적용 도메인

| 도메인 | 소스 예시 | 위키 활용 |
|--------|----------|---------|
| 개인 성장 | 저널, 건강 기록, 독서 노트 | 자기 이해의 구조화된 뷰 |
| 리서치 | 논문, 아티클, 보고서 | 진화하는 논문·테제 |
| 독서 | 챕터 노트 | 캐릭터·테마·플롯 위키 |
| 비즈니스 | Slack, 회의록, 고객 콜 | 항상 최신 내부 위키 |

## 역사적 선행 개념

**Vannevar Bush의 Memex (1945)**: 개인적이고 큐레이션된 지식 저장소. 문서 간 "연상 경로(associative trails)"를 핵심으로 봤다. 그가 풀지 못한 문제 = 누가 유지 관리하는가. LLM이 그 답이다.
