# AI 하네스 & 에이전트 아키텍처

---

## 1. 핵심 원칙

> **재하 = 컨트롤 보유 / AI = 그 구조 안에서 실행**

---

## 2. 하네스의 기본 철학

하네스는 단순한 챗봇 모음이 아니다.

| 역할 | 내용 |
|------|------|
| 분석 | 프로젝트를 분석한다 |
| 선택 | 필요한 에이전트를 선택한다 |
| 분배 | 에이전트 간 역할을 분배한다 |
| 정리 | 개발 지침을 정리한다 |
| 관리 | 작업 흐름을 관리한다 |
| 검증 | 결과물을 검증한다 |
| 기록 | 로그를 남긴다 |
| 문서화 | DB와 문서화를 관리한다 |
| 구조화 | 사용자의 지시를 구조화한다 |
| 절감 | 반복되는 개발 시행착오를 줄인다 |

**핵심 구조 개념**
- 하네스 = 공통 바디 (교체 가능한 머리를 지지)
- 에이전트 = 교체 가능한 머리 (역할별 전문화)
- 프로젝트별로 적합한 프레임워크 선택

## 4. JH-키아누 (jh-keanu) — OMO 대응 시스템

| 항목 | 내용 |
|------|------|
| 비교 대상 | OMO v3.11.0 |
| 규모 | 64파일, 5,158 LOC |
| 스택 | FastAPI + Next.js 14 + Supabase + Claude API |
| 상태 | 구축 완료, 크리티컬 버그 수정 필요 |

**5개 고정 에이전트**

| 에이전트 | 역할 |
|---------|------|
| RESEARCH | 정보 수집 및 조사 |
| PLAN | 계획 수립 |
| FRONTEND | 프론트엔드 구현 |
| BACKEND | 백엔드 구현 |
| QA | 품질 검증 |

**핵심 구성요소**
- 3-tier Hook 엔진
- Ralph Loop (반복 제어 루프)
- Intent Gate (의도 분류 게이트)
- Dynamic Prompt Builder (동적 프롬프트 생성)
- Wave Manager (병렬 실행 파도 관리)

**크리티컬 버그**

문제: `ClaudeClient.send_message()` — sync Anthropic 클라이언트를 async def 내부에서 사용 → FastAPI 이벤트 루프 블로킹

수정 순서:
1. async 클라이언트로 교체
2. IntentGate Haiku API 업그레이드
3. 슬래시 커맨드 추가
4. Supabase Realtime 연동

> ⚠️ JH-하네스와 별개 프로젝트 (혼동 금지)

## 6. JH Agent Hub 3D

| 항목 | 내용 |
|------|------|
| 파일명 | jh-agent-hub-3d.jsx |
| 스택 | React JSX + Three.js |
| 상태 | 프로토타입 완료 |
| 총 에이전트 | 17개 / 4그룹 |

**4개 그룹**

| 그룹 | 용도 |
|------|------|
| 하네스 | 멀티에이전트 조율 |
| 브레인 | 지식 관리 |
| 견적 | 공사비 산출 |
| 판매 | 영업/마케팅 자동화 |

**시각 특징**
- 봇 형태 (머리 + 몸통 + 팔)
- Main = 왕관+안테나, Sub = 구분 없음
- 연결선: flow / sync / data 파티클 라인
- 인터랙션: 그룹 드래그, 카메라 회전, 줌

**다음 단계**
1. 에이전트 추가/삭제 UI
2. 에이전트 간 메시지 애니메이션
3. Next.js + Supabase 풀 프로젝트 전환

**3개 트랙**

|------|------|

**분리 이유**: JH 원칙(승인 기반) vs Playwright(자율 연속 실행) 구조적 충돌
