# Orca IDE 완전정복 | Claude·Codex·Grok 동시에 돌리는 멀티 에이전트 오케스트레이션!

![thumbnail](https://i.ytimg.com/vi/8i0HYZq98VI/maxresdefault.jpg)

## 요약

배움의 달인(김문정)이 SNS에서 뜨고 있는 **Orca(오르카)** 라는 **ADE(Agentic Development Environment, 에이전틱 개발 환경)** 를 소개·시연한다. Cursor·VS Code가 사람이 코딩하려고 터미널을 띄우는 IDE라면, Orca는 **하나의 프로젝트를 워크트리(git worktree)로 나눠 각 워크트리에 서로 다른 LLM CLI(Claude Code·Codex·Grok·Gemini·Antigravity 등)를 분리 배치해 협업시키는 "에이전트 여러 명을 관리하는 작업실"** 이다. 핵심 메시지는 **모델 믹스(model mix)로 비용과 품질을 동시에 최적화**하라는 것 — Codex/Fable 5를 전부 쓰면 좋지만 비용이 크니, 발표자는 **리서치=Grok 4.5, 코딩=Codex GPT 5.6, 디자인·기획·오케스트레이션=Claude Fable 5** 식으로 역할을 나눈다. 실행은 `골(goal)` 명령에 프롬프트를 넣고 작업할 모델들을 지정하는 방식이며, **독립 작업 병렬 실행(여러 모델이 각자 작업 후 최선안 선택)** 과 **단계별 분담(A단계 Codex, B단계 Claude, C단계 Grok)** 두 패턴을 지원한다. Orca 자체는 껍데기라 **각 모델 CLI가 로컬 터미널에 미리 설치·로그인** 돼 있어야 하고, 컴퓨터 유즈(로컬 PC 제어)와 QR 페어링을 통한 **모바일 연결(PC↔모바일 끊김 없는 작업)** 도 제공한다.

## 핵심 개념

- **Orca = ADE(Agentic Development Environment)**: 코딩용 AI 에이전트를 여러 마리 동시에 돌리기 위한 환경. "사람이 코딩하는 IDE"가 아니라 **에이전트 여러 명을 관리하는 하나의 작업실**.
- **워크트리 분리 협업**: 하나의 프로젝트를 **git worktree** 로 나누고, 각 워크트리에 별도의 LLM 도구(Grok/Gemini/Claude Code/Codex)를 붙여 **분리·협업**시킨다.
- **모델 믹스가 핵심 가치**: 고성능 모델만 쓰면 품질은 좋지만 비용 과다. **장단점을 조합(mix)** 하면 효율 + 고품질을 동시에 달성 — 발표자가 강조하는 Orca의 최대 장점.
- **발표자의 역할별 모델 배치(실험 결과)**: 실시간 서치/리서치=**Grok 4.5**, 기획·오케스트레이션·프론트엔드=**Claude Fable 5**, 일상 코딩·테스트 작성/실행=**Grok 4.5로 충분**, 복잡한 코딩·디버깅=**Codex GPT 5.6(X-high)**.
- **`골(goal)` 명령으로 실행**: 명령 프롬프트를 넣고, 일할 모델들을 지정(예: Codex=코딩/GPT 5.6, Claude=디자인/Sonnet 5, Grok=리서치/4.5-high).
- **두 가지 오케스트레이션 패턴**: ① **병렬 독립 실행** — 랜딩 페이지를 Claude·Codex·Grok이 각자 만들어 가장 잘한 결과 선택, ② **단계별 분담** — A단계 Codex, B단계 Claude, C단계 Grok.
- **Orca는 껍데기**: 모델이 돌아가려면 **Claude Code CLI·Grok CLI·Gemini·Codex 등 CLI가 로컬 터미널에 이미 설치**돼 있고, **AI 제공업체 계정 로그인(계정 연결)** 이 완료돼 있어야 한다.
- **개발사 CEO(찐/찡)의 빌드 방식 참고**: Orca 자체를 만들 때 **오케스트레이션·작업 배분을 Grok 4.5** 에 맡기고, 나눠진 일은 **GPT 5.6-high** 가 처리, 완료분을 다시 Grok에 보고해 최종 답변을 받는 흐름. 발표자는 이를 착안해 자기 오케스트레이션을 구성.
- **부가 기능**: **컴퓨터 유즈(로컬 PC 제어)**, **QR 페어링 모바일 앱**(PC↔모바일 동일 프로젝트 끊김 없이 작업), **스킬 CLI/Orca CLI**, 자주 쓰는 도구를 등록하는 **빠른 명령어(quick command)**. 업데이트 주기가 매우 짧아 주기적 확인 권장.

## 언급된 도구/프레임워크

- `Orca(오르카)` — 코딩 AI 에이전트를 여러 마리 동시 구동하는 ADE. 워크트리별 모델 분리 + 오케스트레이션. Mac/앱스토어 다운로드, 다운로드 팩 제공.
- `git worktree` — 하나의 프로젝트를 여러 작업 트리로 나눠 각기 다른 에이전트가 병렬 작업하게 하는 분리 단위.
- `goal(골) 명령` — 프롬프트 + 작업 모델 지정으로 오케스트레이션을 실행하는 Orca 명령.
- `Claude Code CLI / Codex CLI / Grok CLI / Gemini / Antigravity` — Orca 안에서 돌리려면 로컬 터미널에 사전 설치·로그인이 필요한 모델 CLI들.
- `Grok 4.5 (Super Heavy)` — 빠르고 저렴, 실시간 리서치/서치에 강점. 최고 품질은 아니나 일상 코딩엔 충분.
- `Codex GPT 5.6 (X-high)` — 복잡한 코딩·디버깅에 최적.
- `Claude Fable 5 / Sonnet 5` — 기획·오케스트레이션·프론트엔드·디자인에 강점(단, 비쌈).
- `컴퓨터 유즈 / 모바일 QR 페어링 / 스킬 CLI / 빠른 명령어` — Orca 부가 기능(로컬 제어·모바일 연동·스킬 확장·단축 명령).
- 인물: `찐(찡) — Orca 개발사 CEO 추정` — X 포스트에서 Grok 오케스트레이션 + GPT 처리 빌드 방식 공유.

## 우리 시스템 적용 포인트

> JH 브레인 시스템 맥락에서 이 영상의 아이디어를 어떻게 쓸지.
> (Bucky=AI 오케스트레이터, Obsidian Vault=지식두뇌, knowledge-distiller=증류 파이프라인, 목표=1인 창업 레버리지)
- [ ] **역할별 모델 믹스 전략** → Bucky 오케스트레이션에 "리서치=저가·빠른 모델, 복잡 구현·디버깅=고성능 모델, 기획·조율=중상급" 식 비용/품질 라우팅 표를 명시. 이미 우리 생태계가 Claude(구현)·Codex(검수)로 나뉘어 있으므로 **리서치 축(Grok류)** 추가 검토.
- [ ] **워크트리 병렬 독립 실행 패턴** → 동일 태스크를 여러 모델이 각자 수행 후 최선안 선택 → knowledge-distiller의 "정제안 후보 다중 생성 → 최고 품질 선택"이나 견적 파이프라인의 교차검증에 응용 가능.
- [ ] **단계별 분담(A-Codex / B-Claude / C-Grok)** → Bucky 패킷의 `scope`/`next_action`을 단계별 모델 배정과 연결해, 각 단계 산출물을 sidecar에 저장하고 다음 단계로 넘기는 파이프라인 설계.
- [ ] **껍데기+CLI 설치 전제 확인** → 로컬 런타임(집 PC)에서 Claude/Codex CLI 설치·로그인 상태를 세션 시작 체크리스트에 반영(MEMORY의 Bucky CLAUDE_COMMAND 경로 취약성과 연결).
- [ ] **모바일 QR 페어링 원격 제어** → 향후 브레인 시스템의 원격 모니터링/지시 채널(디스코드 봇 토폴로지) 확장 시 참고 레퍼런스.
