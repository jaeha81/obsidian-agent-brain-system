# AI 생성 코드 절반으로 줄이는 5만 스타 스킬, Ponytail 직접 써봤습니다

![thumbnail](https://i.ytimg.com/vi/oaENE_9D95Q/maxresdefault.jpg)

## 요약

출시 2주 만에 GitHub 스타 5만 개를 넘긴 **Ponytail(포니테일)** 스킬을 발표자가 직접 설치해 검증한 실사용 리뷰다. 핵심 문제의식은 코딩 에이전트가 **간단한 요청에도 파일을 여러 개 만들고 단순 로직을 과도하게 복잡화(추상화 레이어·불필요한 의존성 덕지덕지)**하는 현상이다. 발표자는 이것이 AI가 멍청해서가 아니라, **LLM이 사용자 요청에 최대한 친절하게 응답하도록 보상 학습**된 결과라고 진단한다. Ponytail은 "**게으른 시니어 개발자**"라는 컨셉으로 AI에 브레이크를 걸어, 브라우저/표준 라이브러리에 이미 있는 기능을 처음부터 다시 짜지 않게 만든다. 발표자가 동일 프롬프트로 모달·색상 선택기·아코디온·날짜 선택기 4종을 비교한 결과 **기능은 동일하게 동작하면서 코드 라인수가 거의 절반 이하로 감소**(230→100, 197→71, 180→78, 151→61줄)했다. 공식 레포는 최대 94%, 적게는 54%까지 코드가 줄고 **비용은 낮아지고 생성 속도는 빨라진다**고 주장하며, 실측에서도 생성 속도 우위가 확인됐다.

## 핵심 개념

- **문제 = AI Slop(코드 과잉생성)**: 작은 버튼 하나에도 컴포넌트·추상화 레이어·의존성을 붙이고, 날짜 선택기처럼 브라우저 표준에 이미 있는 기능도 처음부터 다시 구현하는 경향.
- **원인 = 과잉 친절**: LLM은 사용자 요청에 최대한 친절하게 응답하도록 보상 학습됨 → 요청하지 않은 것까지 만든다(예: 색상 선택기만 요청했는데 컬러 팔레트까지 자동 구현).
- **Ponytail의 컨셉 = 게으른 시니어 개발자**: 긴머리·둥근 안경·수염의 캐릭터로 "게으르지만 개발 잘하는 시니어"를 형상화. AI에게 "브레이크"를 달아준다.
- **게으른 것이지 소홀한 것이 아니다**: **경계 검증, 데이터 손실 처리, 보안, 접근성**은 절대 코드 압축(생략) 대상이 아니라는 규칙이 명시돼 있음.
- **효과 = 코드량·비용·속도 3중 개선**: 라인수 절반 이하, 코드가 적으니 토큰/비용 절감, 생성 속도도 빨라짐(실측 14초 vs 미적용 32초).
- **한 줄 프롬프트와의 차이**: "YAGNI 원칙을 지켜라"라는 한 줄짜리 프롬프트만 추가해도 어느 정도 효과가 있으나, 실측 라인수는 **Ponytail(100줄) < YAGNI 한 줄 프롬프트(143줄) < 아무것도 미적용(가장 많음)** 순으로 Ponytail이 가장 우수.
- **토큰은 결이 다름**: 토큰 사용량 비교에서는 오히려 YAGNI 한 줄 프롬프트가 낮게 나오는 구간도 있어, 코드 라인수 절감과 토큰 절감이 항상 일치하지는 않음.

## 언급된 도구/프레임워크

- `Ponytail(포니테일)` — AI가 생성하는 코드량을 최소화하는 스킬. 출시 2주 만에 GitHub 스타 5만 개, 독일계 개발자 제작 추정. 최대 94%/최소 54% 코드 절감 주장.
- `YAGNI(야근이)` — "You Aren't Gonna Need It", 필요해지기 전에는 만들지 마라. Ponytail의 첫 번째 적용 규칙.
- `KISS(키스)` — "Keep It Simple", 복잡하게 하지 말고 단순하게. 소프트웨어 3대 원칙 중 하나로 언급.
- `DRY(드라이)` — "Don't Repeat Yourself", 반복하지 말고 재사용. 코드베이스에 이미 있는지 검증하는 규칙과 연결.
- `풀 / 라이트 / 울트라 모드` — Ponytail 강도 3단계. 기본값은 풀(YAGNI 원칙 기본 준수), 약하게는 라이트, 강하게는 울트라.
- 설치 대상: `클로드 코드`, `코덱스(Codex)`, `커서(Cursor)` 등 대부분의 코딩 에이전트 지원. 클로드 코드는 플러그인 추가 후 설치.

### Ponytail 7단계 동작 원리

1. **YAGNI** — 필요해지기 전에는 만들지 않는다.
2. **코드베이스 검증** — 그 기능이 이미 있는지 없는지 확인(DRY).
3. **표준 라이브러리 검증** — 표준 라이브러리로 구현 가능한지(핵심).
4. **네이티브 플랫폼 검증** — 브라우저/네이티브 플랫폼이 원래 해 주는 건 아닌지(핵심). 3·4번에서 대부분의 코드가 절약됨.
5. **기존 의존성 활용** — 이미 설치된 의존성이 있으면 그것을 활용.
6. **한 줄 우선** — 한 줄로 끝낼 수 있으면 웬만하면 한 줄로.
7. **최소 직접 구현** — 정 안 되면 마지막으로 최소한으로 직접 구현.

- [ ] **Ponytail 7단계 = Karpathy "단순함 우선"의 실행 체크리스트** → 프로젝트 CLAUDE.md의 "요청하지 않은 기능·추상화 금지" 규칙을 "표준 라이브러리/네이티브가 하는가 → 기존 의존성 있는가 → 한 줄로 되는가" 순서의 구체 게이트로 보강.
- [ ] **"게으른 것이지 소홀한 것이 아니다" 예외 목록**(경계 검증·데이터 손실·보안·접근성은 압축 금지) → 코드 압축/단순화 지시를 낼 때 이 4개 항목은 항상 예외로 명시하도록 Bucky 패킷 constraints에 표준 문구 추가.
- [ ] **Ponytail 스킬 실도입 검토** → 브레인 시스템 scripts 유지보수 작업에서 클로드 코드 코드량이 과했던 경험과 정합. 플러그인 설치 후 라이트/풀 모드로 estimation·distiller 스크립트 수정 시 AI Slop 억제 실험.
- [ ] **"코드 라인수 절감 ≠ 토큰 절감" 교훈** → 완료 보고 시 "라인수 감소"만으로 개선을 주장하지 말고, 토큰/비용은 별도 측정. Bucky `done_when`에 라인수와 토큰을 분리 지표로 기록.
