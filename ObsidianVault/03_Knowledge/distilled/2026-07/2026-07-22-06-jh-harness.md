> JH-하네스(멀티하네스 아키텍처)의 정본 — "재하가 통제하고 AI는 그 구조 안에서 실행한다"는 핵심 원칙, 비판 세션에서 도출된 3대 설계 문제와 해결책, 하네스 4대 구성요소, 그리고 JH 에이전트 생태계(키아누·브레인시스템·Agent Hub 3D·playwright-agents)의 오케스트레이션 방식 비교를 통합 정리 (2026-04 시점).

## 핵심 인사이트

- **핵심 원칙(불변)**: "재하가 통제한다. AI는 그 구조 안에서 실행한다." 단순 개발 원칙이 아니라 JH 브랜드의 철학적 기반 — AI 자율성에 의존하지 않고, 명시적 승인 없이 에이전트 실행 없음, 제어권은 항상 재하에게.
- **JH-하네스 상태**: 아키텍처 설계 단계(2026-04 시점). **JH-키아누와 별개 프로젝트 — 혼동 금지** (키아누=5개 고정 에이전트·구현 완료, 하네스=동적 구조·설계 중). EstimateAI 대회 출품 버전에 멀티하네스 적용 승인 → 실제 검증 케이스.
- **하네스 개념**: 하네스 = 공통 바디(교체 가능한 머리를 지지), 에이전트 = 교체 가능한 머리(역할별 전문화). 각 하네스는 프로젝트별 독립 작동, 하네스 간 통신은 명시적 인터페이스만, 재하가 하네스 레벨에서 통제. 인터페이스 설계에 `approval_required = True` 기본값 — 승인 없으면 `PermissionError`.
- **비판 세션 3대 설계 문제 → 해결책**: ① YAML 스키마는 선언일 뿐 런타임 동작 강제 불가 → 코드 레벨 구현으로 보장. ② 파일 기반 로그는 삭제 가능해 감사 불가 → Supabase DB 기반 불변 로그. ③ Supervisor가 판단+실행 혼재 시 통제권 상실 → 판단(Supervisor)과 실행(Executor) 명확 분리.
- **하네스 4대 구성요소**: ① 메인 오케스트레이터(요구 분석·프레임워크 선택·계획·업무 분장·품질 관리·보고) ② 전문 에이전트 12종(Frontend/Backend/DB/Security/UI-UX/API/Automation/Documentation/Test/Deploy/Monetization/Interior Domain) ③ 검증 에이전트(코드·보안·법적 위험·UX·성능·구현 가능성 검토 + 과도 기능 제거·MVP 기준 검증) ④ 기록 시스템(개발 로그·프롬프트·오류·결정사항·체크포인트 → GitHub/Obsidian/DB 연동).
- **도입 보류 기술**(현 규모에 과도, 동시 프로젝트 5개 이상 시 재검토): Hermes Agent, Scope Tag(DEV/OPS). 불필요 판정: Skill 자동추출, 컨텍스트 압축, 세션 검색.
- **JH-브레인시스템**(활성 핵심): Obsidian 기반 세컨드 브레인. LangGraph 8-node StateGraph 오케스트레이터, Express.js + Vanilla JS + Google Sheets/Drive + Obsidian. 에이전트 4종 = 므네메(오케스트레이터, Opus) / 므네모시네(Obsidian 기록) / 스카우트(정보 수집) / 빌더(구현). 승인 포인트는 단 하나 — AWAITING 노드에서 "구현해" 패턴 매칭.
- **JH Agent Hub 3D**: jh-agent-hub-3d.jsx, React + Three.js 프로토타입 완료. 17개 에이전트 / 4그룹(하네스·브레인·견적·판매), flow/sync/data 파티클 연결선. 다음 단계: 에이전트 추가/삭제 UI → 메시지 애니메이션 → Next.js + Supabase 풀 전환.
- **jh-playwright-agents**: 13개 에이전트 / 27개 파일, 3트랙(DEV inspection / Business automation / Pre-deploy check). JH 글로벌 원칙과 **완전 분리** — JH 원칙(승인 기반)과 Playwright(자율 연속 실행)의 구조적 충돌 때문.
- **오케스트레이션 방식 비교**: JH-키아누(FastAPI + Ralph Loop, OMO 대응) / JH-브레인시스템(LangGraph 8-node, Obsidian 연동) / JH-EstimateAI(SCANNER→REPORTER 5-agent 파이프라인) / jh-playwright-agents(자율 연속 실행, JH 원칙과 분리).
