# 클로드 코드가 혼자 100명처럼 일하는 법 | 현업이 정리한 Opus 4.8 Dynamic Workflow 소개

![thumbnail](https://i.ytimg.com/vi/nrCln3sWXqU/maxresdefault.jpg)

## 요약
Claude Code에 새로 들어온 Dynamic Workflow(동적 워크플로우)를 중심으로 Subagent, Agent Teams, Agent View까지 이어지는 멀티 에이전트 구조를 하나의 기준으로 정리한다. 그 기준은 "작업 계획이 아니라, 에이전트를 조율하는 계획을 누가 들고 있는가"이다. 메인 세션은 작고 소중한 자원이라 대화가 쌓이면 컨텍스트가 폭발하고 압축을 거치며 중요 정보가 유실된다는 문제에서 출발해, 이를 컨텍스트 격리로 해결하는 Subagent를 설명하고, Subagent를 대규모로 병렬 조율하기 위해 자바스크립트 코드가 오케스트레이션을 맡는 Dynamic Workflow로 발전하는 흐름을 그린다.

## 핵심 개념
- **분해 기준 = 조율 계획의 소유자**: Subagent·Dynamic Workflow·Agent Teams를 구분하는 단 하나의 질문은 "에이전트를 조율하는 계획을 누가 들고 있는가".
- **메인 세션은 희소 자원**: 대화가 쌓이면 컨텍스트가 차오르고, 압축이 일어나면 기대와 달리 중요 내용이 남지 않는 경우가 많다. 계획 1~10을 메인에서 다 수행하면 첫 작업만에 컨텍스트가 폭발.
- **Subagent = 컨텍스트 격리 워커**: 메인이 계획(1~10)만 들고 작업만 위임, 무거운 컨텍스트(소스 탐색·문서·코드작성)는 서브가 따로 보유하고 결과만 반환. 가장 중요한 가치는 "컨텍스트 격리".
- **도구 제한 + 도메인 특화**: 새 세션에 도구를 조금만 주면 그 도구만 쓴다 → 안전. 도메인 특화 시스템 프롬프트로 특정 작업에 최적화 → 작은 모델도 충분.
- **내장 서브에이전트**: `Explorer`(코드베이스 검색·분석, 가장 작은 Haiku 모델), `Plan`(계획 수립, 메인 모델 상속 — 계획은 고려사항이 많아 똑똑한 모델 필요), `General Purpose`(탐색+수정, 보통 Sonnet 상속).
- **Subagent의 한계**: YAML 프론트매터 마크다운으로 정의(스킬과 동일 형식). 다른 서브에이전트를 생성 못 함(중첩 불가) → 메인 세션이 항상 오케스트레이션해야 실행됨.
- **Dynamic Workflow**: 대규모 서브에이전트를 조율하는 자바스크립트 코드. 메인은 "이 계획대로 워크플로우 실행하고 보고해"만 지시. 상태(루프·분기·중간결과)가 스크립트 변수에 "코드에 산다". 메인 컨텍스트엔 최종 답변만 남지만 중간 결과는 확인 가능.
- **규모·격리 확장**: 코드가 관리를 위임받아 대규모 노출 가능(단 Claude Code 내 동시 실행 최대 16개). 코드로 만들었기에 Subagent가 Subagent를 못 만드는 제약도 돌파 가능.
- **동적 제어(키포인트)**: 에이전트별 모델 인텔리전스 라우팅(가벼운 작업=Haiku, 무거운 작업=Opus), 같은 파일 충돌 방지를 위해 Git Worktree로 코드까지 격리 → 결과가 별로면 워크트리를 지우면 끝.

## 언급된 도구/프레임워크
- `Claude Code Dynamic Workflow` — 대규모 서브에이전트를 자바스크립트 코드로 조율하는 신기능. 동시 최대 16개.
- `Subagent` — 컨텍스트 격리된 전문 워커. YAML 프론트매터 마크다운으로 정의.
- `Explorer / Plan / General Purpose` — 내장 서브에이전트(각각 Haiku / 메인모델 상속 / Sonnet 상속).
- `Agent Teams / Agent View` — 멀티 에이전트 구조의 상위 개념(조율/관찰).
- `Git Worktree` — 다중 에이전트의 파일 충돌 방지용 코드 격리. 결과 폐기 시 워크트리 삭제.
- `Opus 4.8 / Haiku / Sonnet` — 작업 난이도별 모델 라우팅 대상.
